# jr-exe-spec

Skill to implement approved technical specs. Receives a `.md` spec, builds an explicit execution plan, presents it for approval, executes changes with full traceability, and updates the spec lifecycle.

---

## Step 0 — Validate input

If the user ran `/jr-exe-spec` **without attaching a `.md` file**, respond in their language:
> [Ask them to share the approved spec: `/jr-exe-spec @specs/feature-name.md`]

Also read `PROJECT.md` for stack and conventions.

---

## Step 1 — Read and validate the spec

1. Read the complete spec `.md`.
2. Verify the spec status:
   - If `Status: Implemented` → warn in user's language: "This spec was already implemented. Do you want to re-implement it or is this a new iteration?"
   - If `Status: Draft` → correct, continue.
3. Verify the spec is ready for execution:
   - Does it have **Functional Requirements** with acceptance criteria?
   - Does it have a **Technical Design** section?
   - Are there `[PENDING]` or `[TBD]` items in critical sections (FRs, Technical Design, APIs)?
4. If critical `[PENDING]` / `[TBD]` items exist, **stop** and report in user's language.
5. Verify specs declared in **Dependencies > Specs that must run first**. If any don't have `Status: Implemented`, warn in user's language.
6. Scan the project:
   - Detected stack
   - Relevant directory structure
   - Files the spec mentions: do they exist? What state are they in?
   - Project conventions

Consult `references/stack-patterns.md` for detected stack patterns.

---

## Step 2 — Build the execution plan

Present the plan to the user in their conversation language:

```
[Plan title — feature name]
[Detected stack]
[Files to create: X | Files to modify: Y | Dependencies to install: Z]

[PHASE 1: Name]
  - [ ] 1.1 [Concrete action] → `path/file.ext` [CREATE]
  - [ ] 1.2 [Concrete action] → `path/file.ext` [MODIFY]

[PHASE 2: Name]
  - [ ] 2.1 [Concrete action] → `path/file.ext` [CREATE]

[PHASE N: Dependencies (if applicable)]
  - [ ] N.1 Install: `[package-manager] add [package@version]`

[PHASE N+1: Suggested manual verification]
  - [ ] [What to verify to confirm the feature works]

[Ask for approval before executing anything]
```

**Plan rules:**
- Each item is an atomic and verifiable action.
- Mandatory order: types/interfaces → models/schemas → services → UI/controllers/routes → styles → config/env.
- Config files with sensitive data (`.env`, secrets) go as `.env.example` with empty keys.

---

## Step 3 — Wait for approval

**Do not execute anything until receiving explicit approval.**

If the user requests adjustments, modify the plan and present it again. Only proceed with "yes", "go ahead", "execute" or equivalent.

---

## Step 4 — Execute the plan

Phase by phase, in order. For each item:

1. **Announce** in user's language: `▶ 1.1 Creating \`src/services/feature.ts\`...`
2. **Execute** the change.
3. **Add traceability** in each file created or significantly modified:
   - JS/TS: `// spec: specs/feature-name.md`
   - PHP: `// spec: specs/feature-name.md`
   - Python: `# spec: specs/feature-name.md`
   - CSS/SCSS: `/* spec: specs/feature-name.md */`
   - Only in files **created by this spec** or code blocks **introduced by this spec**.
4. **Confirm** in user's language: `✓ 1.1 Done`

**If something unexpected is found**, pause and report in user's language before continuing.

**Code writing principles:**
- Follow existing project conventions (naming, import structure, formatting).
- Implement exactly what the spec defines, no more, no less.
- If the spec says "reuse X component/service", find it and reuse it.
- Consult `references/stack-patterns.md` for convention decisions.

---

## Step 5 — Final report and spec update

### Report (in user's conversation language)

```
[Completion message — feature name]

[Files changed table: file, action, description]
[Dependencies installed table: package, version, reason]
[Traceability note]
[How to verify — concrete steps]
[Decisions made during execution]
[Next step: /jr-verify-spec]
```

### Spec update

Open `specs/feature-name.md` and apply:

1. Change `**Status:** Draft` → `**Status:** Implemented`
2. Add `## Affected Files` section (before History):

```markdown
## Affected Files

| File | Action | Description |
|---|---|---|
| `path/file.ext` | CREATED | [what it does] |
| `path/file.ext` | MODIFIED | [what changed] |
```

3. Add entry in `## History`:

```markdown
| 1.1 | YYYY-MM-DD | Implemented | jr-exe-spec — [one-line summary] |
```

---

## Special cases

**File exists with different content than expected:**
Pause. Show expected vs actual diff. Options: overwrite / merge / skip.

**Dependency not available:**
Pause. Report what's missing. Suggest alternative if exists. Wait for decision.

**Spec mentions tests but project has no testing setup:**
Notify at the start. Don't block execution; add as note in final report.

**`.env` / secrets files:**
Never write real values. Create `.env.example` with empty keys and document what goes in each.

**Very large spec (more than 15 items in the plan):**
Before executing, suggest splitting execution into sessions by phase.
